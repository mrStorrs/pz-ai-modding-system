# Project Zomboid B42.19 Codex Instructions

This file is the generic B42.19 compatibility playbook for this workspace. It should read like "if you see X, do Y" and "mods of this type should follow this standard." Specific historical mod fixes belong in `zomboid-update-notes.md` and `zomboid-update-notes.archive.md`.

Read this before changing, installing, debugging, or reviewing Project Zomboid mods for this setup.

## Core Rules

- Do not start Project Zomboid unless the user explicitly asks Codex to launch it again.
- Fix concrete evidence from the newest log first. Do not sweep every installed mod unless the same known pattern is clearly log-proven across active mods.
- Prefer local code and game-bytecode inspection over guessing. If a Lua/Java API changed, verify it in the game code, vanilla Lua, or a known-working active mod before editing.
- Do not silently edit save data. If a save is no longer useful as a test fixture, ask the user for a new save.
- Preserve save-facing IDs unless the user approves a compatibility break: mod `id=`, module names, item IDs, perk IDs, recipe IDs, body-location IDs, map folder names, and vehicle/script IDs.
- If an updated third-party Workshop payload already supports B42.19 and no intentional local edits must be preserved, install it directly into the live `Zomboid/mods` folder. Do not import it into this project just to make a commit.
- Keep a project repo only for CJS/local mods, intentional local compatibility patches, non-drop-in Workshop payloads, or Workshop payloads that still produce proven errors.
- Commit project-owned/imported compatibility changes with messages shaped like `feat(4.19)/{description}`.
- Project-backed mods should be installed as live symlinks back to `/home/cjstorrs/projects/game-mods/zomboid/<mod folder>`.
- Before any loot, EasyDistro, capsule, pack, wrapper, zombie loot, or distribution edit, read `instructions/easy-distributions.md`.

## First Response Workflow

1. Read `zomboid-update-notes.md` for current save, current loadout, removed mods, and latest known log.
2. Inspect the newest dated debug log and `console.txt`.
3. Identify the first concrete root cause before the wrapper failure. `WorldDictionaryException`, hard crash popups, or load aborts are often wrappers.
4. Map the log source to an active mod ID and active live folder.
5. Decide direct Workshop refresh vs project/import patch before editing.
6. Make the smallest compatibility fix that preserves save-facing IDs.
7. Validate locally, link project-backed mods into live if needed, update active load lists if needed, update anchors if state changed, and commit project-owned/imported fixes.

## Symptom To Action Matrix

### `WorldDictionaryException`

If the log ends with `World loading could not proceed, there are script load errors`, search earlier in the same log for:

- `ScriptManager.Load`
- `FileNotFoundException`
- `InputScript.OnPostWorldDictionaryInit`
- `item not found`
- `ScriptBucket.LoadScripts`
- `SpriteConfigManager.parseEntityScript`

Fix the earlier script/item/entity error. Do not patch the final `WorldDictionaryException` wrapper.

### Lowercase `FileNotFoundException`

If a file exists with mixed case but the log requests a lowercased path:

- Rename the active project/live file to the exact lowercased path.
- Do not add duplicate mixed-case/lowercase copies.
- Check all active `media/scripts/**/*.txt` paths for the same casing problem.
- Relink the project-backed live symlink if needed and verify the exact failing path exists.

If the failing path starts with:

```text
/home/cjstorrs/Zomboid/mods/home/cjstorrs/zomboid/mods/...
```

then install a lowercased appdata/mod-root shim:

```bash
pz-linux-native-compat/install-template-path-shim.sh "<LiveModFolder>"
```

The shim name must be lowercase and the target must be the real live folder.

### Missing Script Items

If `InputScript.OnPostWorldDictionaryInit` says `item not found`:

- Extract every missing full type.
- Search active script files for actual `item <Name>` definitions.
- Confirm those script files are under an active `media/scripts` directory for the active version folder.
- If definitions are inside nested `media/scripts/items/` or `media/scripts/recipes/` and the loader is not seeing them, move active `.txt` files up to `media/scripts/` with `git mv`.
- If the item truly no longer exists, patch the recipe/script reference to a real B42.19 item or remove that alternate input.
- Validate that every previously missing item has a definition in the active script root.

### Duplicate Scripted Sprites Or Entities

If `SpriteConfigManager.parseEntityScript`, `ScriptBucket.LoadScripts`, or `Tile duplicate` appears:

- Find all active files claiming the same sprite/tile/entity row.
- Decide which mod should own the entity based on load order, feature ownership, and UI/recipe dependencies.
- Park duplicate generated entity overrides outside `media/scripts`, for example under `media/scripts.disabled-b4219-...`.
- Do not delete files outright when a narrow park preserves rollback.
- Re-scan active `media/scripts` to confirm the duplicate row is gone.

### `No implementation found` For `hasTag` Or Categories

If the log says a function expected `ItemTag`, `WeaponCategory`, or another typed object but got a string:

- Replace string tag calls with typed constants or resolved registry tags.
- Built-in item tags should use `ItemTag.SOMETHING`.
- Custom tags must be namespaced, registered, and resolved as typed tags.
- Do not register unnamespaced custom tags in the default namespace.
- Replace string weapon category checks with typed `WeaponCategory.*` values.
- If current Workshop already made this migration and there are no local edits, refresh live from Workshop instead of patching locally.

### Java Public Field Assignment Stacktraces

If Lua logs `attempted index of non-table` at a line that assigns a Java public field, such as `javaObject.someField = value`:

- Do not rely on `pcall` around direct public-field assignment; Kahlua can still print stacktraces every time the write is attempted.
- Prefer exposed setter methods when bytecode shows they exist.
- If bytecode confirms no setter and a field write is required, use `getNumClassFields` / `getClassField` and the exact `java.lang.reflect.Field` typed setter, such as `field:setBoolean(object, value)`.
- Cache reflected fields, emit one targeted warning on failure, and verify field names against the active game bytecode. For B42.19 `AttackVars`, current boolean fields include `aimAtFloor`, `closeKill`, and `doShove`; old `bAimAtFloor`-style names are stale.

### AmmoType Passed Where String Is Expected

If inventory count/lookup APIs reject `AmmoType`:

- Treat `weapon:getAmmoType()` and `magazine:getAmmoType()` as `AmmoType` objects.
- Convert to item-key strings with `:getItemKey()` before calling string APIs such as `getCountTypeRecurse`, `getItemCountRecurse`, or `getSomeTypeRecurse`.
- Guard nil ammo types before randomization, tooltip, context-menu, or loot paths.
- Firearm script ammo fields should use registry resources such as `base:bullets_9mm`, not stale fulltypes.

### `HandWeapon.randomizeFirearmAsLoot` Null Ammo

If chunk/item spawn crashes because `HandWeapon:getAmmoType()` is nil:

- Search active weapon script overlays for partial firearm definitions.
- Do not overlay a base firearm with only `ItemType = base:weapon` / `Type = Weapon` and omit ammo metadata.
- Preserve or restore all required firearm fields: ammo type, magazine type, attachment data, condition/ranged fields, and model data.
- Prefer refreshing from current Workshop/vanilla definitions and applying a narrow compatibility patch.

### Trait API Errors

If old code calls `HasTrait("...")`, `hasTrait("...")`, or `getTraits():contains("...")`:

- Replace with `character:hasTrait(CharacterTrait.CONSTANT)`.
- Verify constant names in B42.19 vanilla Lua or bytecode.
- Do not rely on `pcall` around invalid Java/Kahlua trait bindings; it can keep Lua running while still logging errors.
- Guard player/npc objects before applying trait logic.

### Stat API Errors

If old code calls direct stat getters/setters such as `getEndurance()`, `setEndurance()`, `getFatigue()`, `setFatigue()`, `getStress()`, `setStress()`, `getBasicStress()`, or old `BodyDamage` boredom/unhappiness methods:

- Use `player:getStats():get(CharacterStat.X)`.
- Use `:set(CharacterStat.X, value)`, `:add(CharacterStat.X, amount)`, or `:remove(CharacterStat.X, amount)`.
- Clamp with the `CharacterStat` object when appropriate.
- Common B42.19 stats include `ENDURANCE`, `FATIGUE`, `STRESS`, `UNHAPPINESS`, `BOREDOM`, `PANIC`, and `SICKNESS`.

### Moodle API Errors

If old code uses `MoodleType.MAX`, `MoodleType.ToIndex`, `MoodleType.FromIndex`, `getNumMoodles()`, or enum index loops:

- Use named `MoodleType` objects.
- Pass `MoodleType` objects to `Moodles:getMoodleLevel(...)` and `getGoodBadNeutral(...)`.
- Iterate `Registries.MOODLE_TYPE:values()` when all moodles are needed.
- For vanilla moodle display order, mirror vanilla UI behavior instead of assuming registry order.
- Do not use `HashMap:entrySet():iterator()` from Kahlua; it can log stacktraces even inside `pcall`.
- Custom-moodle APIs and vanilla-moodle renderers are not interchangeable. Do not replace a custom moodle framework unless the replacement exposes the same creation/storage/control API.

### `PropertyContainer:Is` Or `:Val` Failures

If tile/object/sprite property code calls `props:Is(...)` or stale `props:Val(...)`:

- Use `props:has(...)` for boolean properties.
- Use `props:get(...)` for property values.
- Guard nil `sprite`, `square`, `object`, and `properties`.
- Verify exact property names against B42.19 vanilla or current Workshop.

### `addItemOnServer` Missing

If old loot/award/replacement code calls `container:addItemOnServer(...)`:

- Keep local `container:AddItem(...)`.
- Capture the returned item.
- Call global `sendAddItemToContainer(container, item)` only when it exists.
- Guard client/server context before sending commands.

### `OnFillContainer` Wrapper Errors

If `Events.OnFillContainer` receives a non-`ItemContainer` object:

- B42.19 can pass `ItemPickerJava$ItemPickerContainer` wrappers.
- Gate real container mutation with `instanceof(container, "ItemContainer")`.
- Do not probe Java userdata methods with dot syntax to discover methods.
- Ignore wrappers before calling `getItems`, `Remove`, `AddItem`, or inventory mutation methods.

### Transfer-All, Context Menu, Or UI Row Errors

If transfer-all or context-menu code crashes on separator rows, nil items, or UI rows:

- Guard every row before reading item/category/script data.
- Skip separator, empty, and non-inventory rows.
- Preserve B42.19 vanilla arguments when wrapping timed/equip actions.
- Do not pre-mutate timed action duration in queue hooks if vanilla already does it during `create()` / `begin()`.

### `BodyLocationGroup.isMultiItem` Or Invisible Zombie Body

If zombie death, corpse inventory, or randomized outfits crash around body locations:

- Inspect active clothing/body-location scripts, not only Lua `OnZombieDead`.
- B42.19 converts raw item visuals to worn items before Lua death hooks can clean them.
- Every `BodyLocation` and `CanBeEquipped` value must resolve to a registered `ItemBodyLocation`.
- Vanilla locations should use `base:*`.
- Custom locations must be namespaced, registered, and added to the Human group before item visuals are converted.
- For clothing visuals, mirror Java behavior: use the created item's body location. For non-clothing, use `canBeEquipped()`.
- If the crash is native nil dereference despite valid mod cleanup attempts, use the tracked native helper. Do not hand-edit the jar.

### `Failed to load asset` Or Worn Clothing Does Not Render

If clothing can be worn but is invisible or body parts disappear:

- Read the exact missing asset path from the log.
- Check the active XML model path and active `models_x` / `models_X` folder.
- If the file exists in current Workshop and live drift is the issue, restore the missing asset directory from Workshop.
- If the referenced old model is parked or incompatible, remap the XML to a close B42.19 vanilla skinned mesh.
- Do not restore parked legacy FBX files wholesale unless the exact asset is proven compatible and active XML should still use it.
- Validate by searching the fresh log for the exact missing asset path.

### `ImportedSkeleton.collectBoneFrames`

If old FBX/X animation files log missing bone names:

- Check whether the engine continues after skipping the channel.
- If the issue is log noise from obsolete animation channels and gameplay still works, use the tracked log-downgrade helper rather than binary model rewrites.
- If animation behavior is actually broken, patch the animation references to B42.19-safe vanilla nodes or current Workshop animations.

### `KahluaTableImpl.save` Stack Overflow

If save/load crashes with repeated `KahluaTableImpl.save(...)`:

- Suspect cyclic tables in saved `modData` or global mod data.
- Do not store controller, module, self, player, item, zombie, body, container, or live game-object tables in saved data.
- Persist flat scalar keys only.
- Clear or migrate only the mod-owned save keys required by the fix; do not do broad save cleanup without user approval.

### `ReturnValues.put(ReturnValues.java:61)`

If Java/Kahlua logs `Cannot assign field "callFrame" because "a" is null` from `ReturnValues.put`:

- Recognize this as a B42.19 Kahlua pool/re-entrancy issue when Java called from Lua re-enters Lua before returning.
- Use `pz-linux-native-compat/install-kahlua-returnvalues-null-guard.sh`.
- Do not wrap every Lua caller in `pcall` as the primary fix.

### `ItemPickInfo -> cannot get ID for container`

If native item-pick logs cannot get an ID for active vehicle/container names:

- Lua registration may be too early or unavailable because native preprocessing clears and rebuilds the map.
- Add names through the tracked native helper that extends the built-in `vehicle_containers` array.
- Keep the helper idempotent and verify the jar after patching.

### `unknown command:...`

If an "unknown command" log appears:

- Find the logger before assuming vanilla command dispatch failed.
- Catch-all command loggers should only log unknown commands for their own module namespace.
- In single-player, route mod client/server commands locally when networking is not active.

### Map Selector Shows Translation Paths

If map selection shows `See media/lua/shared/Translate/...`:

- Resolve the referenced title/description text.
- Replace `title=` and `description=` in `map.info` with literal display text.
- Keep `description=` on one line using `<LINE>` markers.
- Verify the active map folder name and `lots=` entries still match.

### Sandbox, Trait, Item, Tooltip, Or Recipe Text Shows Raw Keys

If UI shows raw translation keys or file paths:

- Prefer active-loadout generated B42 JSON coverage instead of hand-patching every mod.
- Generate from active live roots only.
- Preserve load-order precedence for duplicate keys.
- Fold legacy categories into canonical B42 JSON categories.
- Trait XP boost rows need `IG_UI` keys such as `IGUI_perks_<PerkName>`.
- Sandbox options need `Sandbox.json` keys or fallback labels generated from `media/sandbox-options.txt`.

### Lua Reload On Load

If Lua reloads unexpectedly during save load:

- Compare latest save `mods.txt` with `mods/default.txt`.
- If the user has asked for preset sync, compare `b42-4` too.
- Normalize exact IDs and order.
- Do not use display names or folder names in load lists.

## Standards By Mod Type

### B42 Mod Packaging Standard

- Live mods must be direct children of `/media/cjstorrs/windows/Users/cjsto/Zomboid/mods`.
- Project repos live under `/home/cjstorrs/projects/game-mods/zomboid/<mod folder>`.
- Project-backed live entries should be symlinks whose names match the project folder casing exactly. Real copied project-backed live folders are stale and should be replaced by symlinks.
- Active B42 folders must have valid `mod.info`, `poster`, and `icon` paths where declared.
- New CJS B42 mods should include `42/` and `Common/`; add `.gitkeep` for empty required directories.
- Match existing casing in existing mods, but make active script file paths lowercase when B42.19 Linux requests lowercase paths.
- Do not leave accidental nesting such as `Contents/mods/<mod>/42/mod.info` inside live `mods`.

### Load List Standard

- Use exact `id=` values from active `mod.info`.
- Update latest save and default together unless the user explicitly narrows scope.
- Update `b42-4` when the user has asked for preset sync.
- Keep order stable unless fixing an order-dependent issue.
- After edits, verify every intended ID appears exactly once and removed IDs are absent.
- Record parked/removed/restored decisions in `disabled-mods.md` when mods are disabled or removed.

### Direct Workshop Standard

Use direct Workshop install when:

- The mod is third-party.
- Workshop cache has a current B42.19-compatible payload.
- The payload keeps the required `id=`.
- There are no intentional local edits to preserve.
- The needed fix is not a real code/content patch beyond live-only structure normalization.

Project/import patch when:

- The mod is CJS/local.
- Local functionality must be preserved.
- Workshop is absent or not a drop-in.
- Fresh logs prove Workshop still breaks.
- The user explicitly wants local behavior kept.

### Clothing Mod Standard

Clothing mods should follow this B42.19 standard:

- Script clothing items should use B42 item type declarations that produce clothing inventory items, normally `ItemType = base:clothing`.
- `BodyLocation` must resolve through `ResourceLocation.of(...)` to an `ItemBodyLocation`.
- Vanilla body locations should be written as `base:*` resource IDs when B42.19 requires resource IDs.
- Custom body locations must be namespaced, registered, and added to the Human group before clothing/item visuals are converted.
- `ClothingItem` values must point to existing active XML files.
- XML `<m_MaleModel>` and `<m_FemaleModel>` paths must point to existing active model assets.
- If a model is parked or incompatible, remap XML to a safe B42.19 skinned mesh rather than restoring legacy FBXs blindly.
- Keep active root and versioned clothing XML/script copies consistent when both are loaded by the mod.
- If items can be worn but do not display, inspect model assets and XML paths before changing wear actions.
- If clicking `Wear` does nothing, check item type/category first, then global transfer/wear wrappers.

### Weapon And Firearm Standard

Weapon mods should follow this B42.19 standard:

- Use B42 item type/resource syntax consistently; do not mix partial old `Type = Weapon` overlays with incomplete new `ItemType` data.
- Preserve all required weapon metadata when overriding vanilla weapons.
- Firearms must have valid `AmmoType` resources, magazine definitions where applicable, and attachment definitions only when those items exist.
- Code paths that count ammo must convert `AmmoType` objects to item-key strings.
- Context-menu and tooltip code must guard nil ammo, nil magazine, and missing attachment items.
- Wrappers around vanilla equip/reload/firearm actions must preserve all B42.19 arguments.
- Custom projectile weapons should keep their model/sprite namespace prefixes unless asset lookup proves the prefix is wrong.
- Test custom weapons separately from vanilla firearms. A vanilla pistol working does not prove a bow/projectile weapon is configured correctly.

### Map Mod Standard

Map mods should follow this B42.19 standard:

- `map.info` must have stable literal `title=` and `description=` values for map selector display.
- `description=` should stay one line, using `<LINE>` if needed.
- Map folder casing matters. Verify the runtime map token, folder name, and `lots=` entries.
- Do not disable map mods from an existing save just to silence metadata validation logs.
- Do not hand-edit `map_meta.bin` or binary map files without a narrow evidence-backed tool.
- Duplicate room/meta/building validation logs from layered maps are not automatically fatal.
- If the map is unavailable in a missing-mod/modal path, inspect `MapGroups` behavior: folder presence, `map.info`, `lots=`, dependencies, and active mod ID.

### Translation Standard

Translation fixes should follow this standard:

- Prefer generated B42 JSON compatibility for active legacy `*_EN.txt` categories.
- Generate from active live roots only, not every installed mod.
- Preserve load-order precedence for duplicate keys.
- Support category aliases: `Itemname`, `Item_Name`, and `Items` go to `ItemName`; lowercase `sandbox` goes to `Sandbox`.
- Cover common active categories: `Sandbox`, `UI`, `IG_UI`, `ItemName`, `Tooltip`, `ContextMenu`, `Recipes`, `Moodles`, `Recorded_Media`, and related categories.
- Validate every generated JSON file.
- Fix map selector text in `map.info`; do not rely on map selector resolving legacy title path placeholders.

### Crafting, Capsules, Packs, And Loot Standard

Craft/loot mods should follow this standard:

- Concrete craft inputs should use B42 bracketed item-list syntax, such as `item 4 [Base.Item] flags[...]`.
- Output-style bare item syntax should not be used for concrete inputs.
- Capsule/pack/wrapper recipes should consume the wrapper input with `mode:destroy`.
- Distribution systems should spawn wrapper items when the gameplay expectation is opening/unpacking to receive contents.
- Do not add contained rare items directly to loot tables unless that is explicitly intended.
- Craft `OnCreate` callbacks should avoid returning Java userdata into fragile UI/dropdown paths unless vanilla uses that pattern.
- For EasyDistro changes, apply `instructions/easy-distributions.md` before editing.

### Moodle And Trait Mod Standard

Trait/moodle mods should follow this standard:

- Trait checks use typed `CharacterTrait` constants.
- Stat effects use `CharacterStat` APIs.
- Custom moodle creation/storage/control requires a real custom-moodle API; a vanilla moodle renderer is not a replacement.
- Moodle rendering should use B42 `MoodleType` objects and registry values.
- Per-frame trait effects must be bounded and must not repeatedly mutate active timed-action durations unless that is the only safe hook.
- Tooltip text and trait XP boost rows need B42 JSON translation coverage, especially `IG_UI`.

### UI, Context Menu, And Timed Action Standard

UI/action mods should follow this standard:

- Preserve B42.19 vanilla action constructor arguments when wrapping actions.
- Avoid monkey-patching queue insertion unless vanilla has no suitable hook.
- Guard nil item, nil player, nil square, nil container, separator row, and non-inventory UI row cases.
- Do not use debug-only reflection helpers in runtime UI or update loops.
- Keep context-menu logic typed: `ItemTag`, `ItemType`, `WeaponCategory`, `AmmoType:getItemKey()`, and `ItemBodyLocation` where required.

### Native Patch Standard

Native/game-jar patches are last-resort compatibility shims.

- Use tracked helpers under `pz-linux-native-compat`.
- Do not hand-edit jars.
- Keep helpers idempotent and dry-run-capable when possible.
- Back up the original jar under the game install `.codex-backups`.
- Run `bash -n` on helpers.
- Validate jar integrity with `python3 -m zipfile -t` or equivalent.
- Use `javap` to confirm the installed bytecode shape.
- Avoid verifier-risky edits unless the helper updates stack maps/line tables correctly.

## Validation Checklist

For code/script changes:

- Run `luac5.1 -p` on changed Lua files when possible.
- Use `lua5.1 loadfile` only for syntax checks; do not execute PZ Lua outside the game.
- Run `git diff --check`; use CRLF-aware settings if the repo preserves CRLF.
- Search for stale exact patterns you meant to remove.
- For project-backed installs, verify the live entry is a symlink with the same casing as the project folder and `readlink -f` resolves to the project mod folder.
- For direct Workshop copied installs, compare Workshop source and live touched files byte-for-byte when applicable.
- Verify active `mod.info`, `poster`, and `icon` paths exist.
- Verify load-list IDs appear exactly once where intended.
- Run the Zomboid review gate for code, script, asset-reference, packaging, sandbox, mod-list, or live-install changes.
- State the exact in-game validation still needed.

For direct Workshop installs:

- Verify Workshop source exists and has the intended `id=`.
- Back up old live folder.
- Copy Workshop payload to a direct child of live `mods`.
- Verify live and Workshop compare cleanly, except intentional live-only path normalization.
- Do not create a project commit.

For asset fixes:

- Trace the exact logged asset path.
- Verify active XML/script references.
- Verify the referenced file exists in the active live folder.
- Prefer current Workshop assets when live drift is the issue.
- Prefer XML remaps when old assets are incompatible.
- Retest or ask the user to retest the exact item/model.

For load-list edits:

- Back up edited files.
- Preserve order unless the fix is order-specific.
- Verify default/save/preset scope matches the user's latest instruction.
- Run a parsed comparison when lists should match.
