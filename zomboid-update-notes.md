## B42.19 Fix Patterns Learned

- Player endurance now uses the B42 stat enum API: read with `player:getStats():get(CharacterStat.ENDURANCE)` and write with `player:getStats():set(CharacterStat.ENDURANCE, CharacterStat.ENDURANCE:clamp(value))`. Old `getEndurance()`/`setEndurance()` calls can throw `Object tried to call nil`.
- Player stat deltas should use `player:getStats():add(CharacterStat.X, amount)` / `remove` / `set`; old direct setters such as `setStress(...)`, old `getBasicStress()`, and old `BodyDamage` unhappiness setters are not reliable in B42.19. Vanilla `ISInventoryTransferAction` uses `CharacterStat.UNHAPPINESS` and `CharacterStat.STRESS`.
- Do not use debug-only Java reflection helpers such as `getNumClassFields`, `getClassField`, or `getClassFieldVal` in runtime mod code. In B42.19 non-debug runtime paths they can throw `Not in debug`; prefer verified public methods/fields such as `zombie:getSpeedType()` and `zombie.cognition`.
- B42.19 on Linux lowercases mod script load paths. For mixed-case mod IDs or script filenames, make `42/media/scripts/**/*.txt` filenames lowercase in the project/live mod, and ensure lowercased loader-root shims exist under both normal appdata and helper cache appdata when the mod is active.
- B42.19 `Events.OnFillContainer` can pass `ItemPickerJava$ItemPickerContainer` wrappers, not only `ItemContainer`. Do not probe Java userdata methods with dot syntax; gate with `instanceof(container, "ItemContainer")` before `getItems`, `Remove`, or `AddItem`.
- B42.19 body locations are resource IDs. Vanilla locations need `base:*` values like `base:pantsextra` or `base:ankleholster`; custom locations should be registered and used with a namespaced ID. Legacy short names such as `PantsExtra` in `BodyLocation` / `CanBeEquipped` can resolve nil and crash `BodyLocationGroup.isMultiItem`.
- `InventoryItem:getBodyLocation()` now returns an `ItemBodyLocation` object. Do not call string methods such as `bodyLocation:contains(...)` on it. For broad custom-family checks, compare `tostring(bodyLocation)` against the registered namespaced ID prefix, or compare directly against known `ItemBodyLocation` constants/registered objects.
- B42.19 zombie death inventory conversion uses the raw `IsoZombie.itemVisuals` field. If `WornItems.setFromItemVisuals()` gets an item visual whose clothing `BodyLocation` or non-clothing `CanBeEquipped` resolves nil, `WornItems.setItem()` can crash before any Lua `OnZombieDead` handler runs. The installed native guard returns early when `location` is null; do not rely on `OnZombieDead` or `zombie:getItemVisuals()` alone for this crash path.
- `ItemPickInfo -> cannot get ID for container: ...` is produced by Java `ItemPickInfo.GetPickInfo()` when `ItemConfigurator.GetIdForString(containerType)` returns `-1`. `ItemConfigurator.Preprocess()` clears and rebuilds that table, so early Lua registration can be wiped. For always-needed legacy/container IDs, patching the native `vehicle_containers` initializer is more reliable than a Lua startup hook.
- `ImportedSkeleton.collectBoneFrames` missing-bone lines are emitted when an animation channel references a node name absent from the imported skeleton `boneIndices`. `javap` confirms the engine logs and then skips the channel. For old FBX/X files with obsolete `Body` channels, downgrading that local log to `DebugType.noise` is safer than broad binary model rewrites.
- `unknown command:<module>.<command>` can come from another mod's catch-all command listener, not necessarily vanilla. Search the literal log string before changing the command sender. For `VorpallySauced.weaponUpdate`, the source was Vac Mod Utils, and the correct fix was scoping Vac's unknown-command print to `VacModUtils`.
- B42.19 `Events.OnHitZombie` passes `(zombie, attacker, bodyPart, weapon)`. Do not copy older handlers that treat the first argument as the attacker; that silently disables evidence or gameplay logic.
- B42.19 trait checks should use `player:hasTrait(CharacterTrait.SOME_TRAIT)` instead of legacy `player:HasTrait("StringName")`. Verified by `javap -classpath .../projectzomboid.jar zombie.characters.IsoGameCharacter`, which lists `hasTrait(zombie.scripting.objects.CharacterTrait)` and no `HasTrait(String)`. For Catseye, vanilla night vision is `CharacterTrait.NIGHT_VISION`.
- B42.19 trait collection checks should also avoid `player:getTraits():contains("StringName")`; `CharacterTraits` exposes typed `get/add/remove/set(CharacterTrait)` and the normal Lua-visible path is `player:hasTrait(CharacterTrait.SOME_TRAIT)`.
- B42.19 transfer/timed action code should not pre-adjust action duration in `ISTimedActionQueue:addToQueue`. Vanilla `ISBaseTimedAction:create()` applies `adjustMaxTime` before the action begins; extra queue hooks can double-adjust or mutate handcraft/entity actions at the wrong point in the call chain.
- Inventory `itemslist` rows can be grouped rows or UI separator rows, especially with CleanUI. Before reading item methods such as `getDisplayCategory()`, normalize with `ISInventoryPane.getActualItems(...)` and skip empty rows.
- Craft `OnCreate` callbacks that add an item to the player's inventory should prefer the boolean `ItemContainer:AddItem(fullType, 1.0)` overload when the return value is not needed. Avoid returning Java `InventoryItem` userdata from capsule/wrapper callbacks unless the caller explicitly expects that object.
- B42.19 removed the Lua-visible `ItemContainer:addItemOnServer(...)` method used by older mods after `container:AddItem(...)`. Keep the local `AddItem(...)` call, then call global `sendAddItemToContainer(container, item)` when available if client/server sync is needed. Do not probe Java userdata with dot-style method guards for a replacement method.
- B42.19 animation variables should be read from character/zombie userdata with `GetVariable("name")`; legacy Lua-visible `getVariableString("name")` and `getVariableBoolean("name")` can throw `No implementation found`. Verified by `javap -classpath .../projectzomboid/projectzomboid.jar zombie.characters.IsoGameCharacter`, which lists `GetVariable(java.lang.String)`.
- B42.19 moodles need typed uppercase `MoodleType` constants such as `MoodleType.PANIC`, `MoodleType.ENDURANCE`, and `MoodleType.HAS_A_COLD`; legacy names like `MoodleType.Panic` or dynamic `MoodleType[MoodleID]` can fail. Verified with `javap` on `zombie.scripting.objects.MoodleType`.
- B42.19 vanilla `MoodlesUI` stores per-moodle UI data in a Java `HashMap`, so a Lua overlay that draws by registry index can put the right level texture on the wrong visible moodle. To mirror vanilla ordering from Lua, use `HashMap.new()`, insert the same moodle type objects in registry order, then iterate `ArrayList.new(map:values()):iterator()`. Do not use `HashMap:entrySet():iterator()` in Kahlua; it produced logged stacktraces even inside `pcall`. Do not use debug-only reflection helpers for this.
- B42.19 endurance moodle levels use remaining `CharacterStat.ENDURANCE`, not an "exhaustion percent" value: above `0.75` is hidden/level 0, `0.75..0.5` is level 1, `0.5..0.25` is level 2, `0.25..0.1` is level 3, and `<= 0.1` is level 4. A player at roughly 88% exhausted has about 12% remaining endurance, so the visible Endurance moodle should be level 3 or near level 4.
- B42.19 item tag/category APIs are typed. Use `ItemTag.*` / typed `ItemTag` and `WeaponCategory.*`; string calls such as `item:hasTag("ShowCondition")` or `containsWeaponCategory("Axe")` can throw `No implementation found` / `expected argument of type WeaponCategory, got String`. If the upstream Workshop payload already fixed this, refresh live directly instead of importing.
- Custom B42 item tags used from Lua must be namespaced in scripts and registries, then resolved to typed `ItemTag` objects using the same full ID. Example from Take A Bath: script tags use `TABAS:BathSalt` / `TABAS:Empty`, `registries.lua` registers those full names, and call sites resolve the typed tag before `item:hasTag(tag)`. Do not register unnamespaced custom tags such as `BathSalt`; B42.19 rejects default-namespace custom tags.
- B42.19 container tag lookups are typed too. Calls like `getFirstTagEvalRecurse("Wipeable", ...)` or `getAllTagEvalRecurse("BathSalt", ...)` need a resolved `ItemTag`, not a string.
- B42.19 `HandWeapon:getAmmoType()` returns an `AmmoType` object. APIs that still expect an item key string, such as `ItemContainer:getCountTypeRecurse(String)`, need `weapon:getAmmoType():getItemKey()` first. Verified with `javap` on `zombie.scripting.objects.AmmoType`, which exposes `getItemKey()`.
- `getModFileReader(...)` can return nil for missing mod files even when `createOnNil` is true. Guard before `:readLine()` and let callers create defaults with their existing save path.
- If B42.19 logs `ImportedSkeleton.collectBoneFrames` for old mod FBX/X files and those assets are parked, do not fix later `Failed to load asset` torso warnings by restoring the parked assets. For clothing XML, prefer remapping `<m_MaleModel>` / `<m_FemaleModel>` to a close vanilla B42.19 skinned mesh that already loads, then validate in-game for invisible body parts and asset warnings.

## Fix Patterns Learned

- B42.19 Linux can expose stale case-sensitive script paths. Fix active/loadout-relevant mods only.
- B42.19 `Events.OnFillContainer` can pass `ItemPickerJava$ItemPickerContainer` wrappers, not only `ItemContainer`. Do not probe Java userdata methods with dot syntax; gate with `instanceof(container, "ItemContainer")` before `getItems`, `Remove`, or `AddItem`.
- B42.19 body locations are resource IDs. Vanilla locations need `base:*` values like `base:pantsextra` or `base:ankleholster`; custom locations should be registered and used with a namespaced ID. Legacy short names such as `PantsExtra` in `BodyLocation` / `CanBeEquipped` can resolve nil and crash `BodyLocationGroup.isMultiItem`.
- B42.19 map selector can display literal `See media/lua/shared/Translate/EN/...` paths from `map.info`. For active map mods, resolve the referenced `title.txt`/`description.txt`, then replace `title=`/`description=` in `map.info` with literal display text. Keep `description=` on one line using `<LINE>` markers.
- B42.19 translation JSON compatibility can be handled by an active-loadout CJS compatibility mod when many enabled third-party mods still only ship legacy `*_EN.txt`. Generate flat JSON keys from active roots only, support legacy dotted/spaced keys, preserve load-order precedence for duplicate keys, validate every generated JSON file, and append the CJS compat mod once to latest save/default lists.
- Common legacy category aliases should be folded into canonical B42 categories: `Itemname_EN.txt`, `Item_Name_EN.txt`, and `Items_EN.txt` belong in `ItemName.json`; lowercase `sandbox_EN.txt` belongs in `Sandbox.json`.
- B42.19 trait-list XP boost rows use `PerkFactory.getPerkName(perk)`, so custom skill/perk labels such as Efficiency and Scavenging need active `IG_UI.json` keys like `IGUI_perks_Efficiency` and `IGUI_perks_Scavenging`. If trait labels resolve but the XP/perk line shows a path, extend `cjsB42TranslationCompat` for `IG_UI` before editing individual trait mods.
- `.tools/pz-continue-debug.sh` is the canonical Codex startup path. It starts B42.19 with `-debug`, uses a project-local Java cache parent, clicks `Continue`, waits for `game loading took`, and clicks `Click to Start`. If the load marker times out, it captures diagnostics under `.runtime/pz-diagnostics/`. Keep the default `PZ_CONTINUE_X_PCT=0.13`; `0.18` misses the button.
- B42.19 vanilla generated admin entities can still be malformed. For the piano `CraftRecipe` missing `UiConfig`, use `pz-linux-native-compat/install-vanilla-piano-uiconfig-patch.sh` instead of hand-editing the native install.
- B42.19 moodle types are named static `MoodleType` objects. Do not use old `MoodleType.ToIndex(...)`, `MoodleType.MAX`, `MoodleType.FromIndex(...)`, or enum `:index()` loops. `Moodles:getMoodleLevel(...)` and `Moodles:getGoodBadNeutral(...)` take a `MoodleType` object.
- B42.19 character stats should generally use `player:getStats():get(CharacterStat.X)` / `:set(CharacterStat.X, value)` instead of old direct getters/setters such as `getFatigue()` / `setFatigue()`. Patch active, log-proven sites first.
- B42.19 firearm ammo references should use registry resources such as `base:bullets_9mm`; old fulltypes can trigger `HandWeapon.randomizeFirearmAsLoot` null ammo crashes.
- B42.19 removed or changed several Lua APIs, including old property `:Is(...)` calls and some stats getters.
- `KahluaTableImpl.save(...)` stack overflows usually mean a cyclic table reached save serialization. Do not store controller/self/live game-object tables in saved `modData`.
- `ReturnValues.put(ReturnValues.java:61)` with `Cannot assign field "callFrame" because "a" is null` can be a B42.19 Kahlua pool/re-entrancy bug when a Java method called from Lua re-enters Lua before returning, such as `StartAction()` or Neat Crafting recipe callbacks. Use the tracked native helper `pz-linux-native-compat/install-kahlua-returnvalues-null-guard.sh`; do not paper over each Lua caller with `pcall`.
- B42.19 Linux can lowercase script load paths. If a live file is mixed-case and the log shows a lowercased `FileNotFoundException`, rename the file to the exact lowercased path rather than adding a duplicate.
- B42.19 Linux can also lowercase the entire appdata/mod-root URI during script loading. For normal `$HOME/Zomboid` launches, install lowercased live root shims under `/home/cjstorrs/Zomboid/mods/home/cjstorrs/zomboid/mods/<lowercase-root> -> /home/cjstorrs/Zomboid/mods/<real-root>` using `pz-linux-native-compat/install-template-path-shim.sh`. This is required when the stacktrace path starts with `/home/cjstorrs/Zomboid/mods/home/cjstorrs/zomboid/mods/...`.
- B42.19 native vanilla data can still contain malformed map-zone entries. Do not hand-edit game files without a tracked helper. Use `pz-linux-native-compat/install-vanilla-muldraugh-mannequin-zone-patch.sh` for the current Muldraugh mannequin fix.
- B42 craft-recipe inputs that name concrete items need bracketed item-list syntax such as `item 4 [Base.Item] flags[...]`; bare `item 4 Base.Item` is output-style syntax and can crash script loading as `unknown recipe param`.
